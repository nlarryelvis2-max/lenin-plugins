# lenin-uplink/1 — контракт ручки централизации сессий

Дата: 2026-07-17 · Статус: v1, клиент готов и обкатан на мок-сервере ·
Клиент: `scripts/session_uplink.py` (внутри плагина lenin-uplink) · Мок: `scripts/uplink_mock_server.py`

## 1. Назначение

Каждый Мак с ядром Ленина раз в сутки (или при включении / первом заходе в
Ленин) досылает на центральный сервер **новые байты** сессионных файлов Claude
Code (`~/.claude/projects/**/*.jsonl`) — полную историю взаимодействия
человек↔Ленин — для дальнейшей пост-обработки. Сервер = приёмник и хранилище;
пост-обработка живёт отдельно и контракта не касается.

Принцип: **клиент туп, сервер — источник истины по offset'ам.** Клиент не
знает состояния сервера и не обязан помнить его правильно; сервер каждым
ответом говорит, с какого байта продолжать.

## 2. Идентификация — кто прислал

Три оси, все обязательны:

| Поле | Что это | Пример |
|---|---|---|
| `owner_id` | человек-владелец ядра | `owner1`, `owner2` |
| `core_id` | какое ядро установлено | `lenin-core-1`, `lenin-core-2` |
| `machine_id` | конкретная машина (LocalHostName) | `MacBook-Pro-1` |

Токен выдаётся **на пару (owner_id, core_id)** при раздаче обновы. Сервер
ОБЯЗАН проверять, что owner_id/core_id из запроса соответствуют токену —
заголовки декларативны, истина в токене. Один владелец × несколько машин =
один токен, разные `machine_id`.

## 3. Ручка

```
POST /v1/uplink/sessions
Authorization:    Bearer <token>
Content-Type:     application/json
Content-Encoding: gzip
X-Uplink-Proto:   1
X-Owner-Id:       <owner_id>
X-Core-Id:        <core_id>
X-Machine-Id:     <machine_id>
```

Тело — gzip поверх JSON:

```json
{
  "proto": "lenin-uplink/1",
  "owner_id": "owner1",
  "core_id": "lenin-core-1",
  "machine_id": "MacBook-Pro-1",
  "lenin_version": "core 0.1.3 / uplink 1.1.0",
  "sent_at": "2026-07-17T13:06:42+00:00",
  "chunks": [
    {
      "path": "-Users-owner-projects/abc-123.jsonl",
      "offset": 51200,
      "length": 8192,
      "sha256": "<hex sha256 сырых байт чанка>",
      "b64": "<base64 сырых байт>"
    }
  ]
}
```

Семантика чанка: «в файле `path` начиная с байта `offset` лежат вот эти
`length` байт». Чанк всегда заканчивается на границе строки (`\n`) — клиент
не режет JSONL-строку пополам. `path` — путь относительно
`~/.claude/projects/`, всегда относительный, без `..` (сервер обязан
отклонять иное — path traversal).

Лимиты v1: чанк ≤ 8 МБ сырых байт, батч ≤ 24 МБ сырых байт, за один прогон
клиент шлёт ≤ 200 МБ (несколькими запросами). Сервер должен принимать тело
до ~32 МБ после распаковки.

## 4. Ответы

**200 — принято (полностью или частично):**

```json
{ "accepted": true,
  "files": { "<path>": <next_offset>, ... } }
```

`next_offset` — сколько байт этого файла сервер теперь имеет. Клиент ставит
свой манифест ровно в это значение. Этим одним полем решаются все три случая:

| Случай | Сервер делает | `next_offset` |
|---|---|---|
| `offset == текущий размер у сервера` | append байт | `offset + length` |
| `offset <` размера (ретрай дубля) | ничего (идемпотентно) | текущий размер |
| `offset >` размера (дыра: у сервера меньше) | ничего | текущий размер — клиент дошлёт недостающее следующим прогоном |

**Ошибки:** `401` неверный токен · `403` токен отозван (см. §6) · `400`
битое тело/абсолютный path · `413` превышен лимит · `429`/`5xx` — временно.
На любую ошибку клиент НЕ двигает манифест и молча повторит всё в следующий
прогон (launchd завтра / SessionStart). Отдельной ретрай-логики не нужно —
идемпотентность её заменяет.

## 5. Хранение (минимальные требования)

- Append-only, сырые байты как пришли: `<owner_id>/<machine_id>/<path>`.
  Никакого парсинга при приёме — сессии восстанавливаются байт-в-байт.
- Перед append сверить `sha256` чанка; не сошёлся → `400`, offset не двигать.
- Журнал приёмов (когда/кто/сколько) — для аудита и отладки «почему у клиента
  не отправлено N МБ».

## 6. Безопасность и этика (канон data-exchange-ethics)

- Только TLS на реальном сервере (мок — localhost).
- Сырые транскрипты = **personal_special**: у других владельцев в
  сессиях личное владельцев ядер. V1 подключает ТОЛЬКО машины Фила. Флот —
  после явного consent владельца; consent фиксируется при выдаче токена.
- Отзыв (right to erasure / FROZEN): сервер умеет пометить токен отозванным →
  `403`; клиент при `403` выставляет `enabled: false` в своём конфиге
  (v1: вручную; v2: автоматически).
- Токен в `~/.claude/lenin_uplink/config.json` (chmod 600 — на совести
  установщика обновы).

## 7. Что несёт обнова ядра (клиентская сторона, уже готово)

| Компонент | Роль |
|---|---|
| `session_uplink.py` | клиент: скан → инкремент по манифесту → POST; один файл, stdlib, без зависимостей |
| `session_uplink_check.py` (SessionStart-хук) | заход в Ленин → если last_ok >24ч → фоновый прогон |
| `com.lenin.session-uplink` (launchd) | ежедневно 08:10 + RunAtLoad (Мак включили) |
| `config.json` | endpoint, token, owner_id, core_id — единственное место настройки |

Установка на машину: `python3 session_uplink.py --install-launchd` + хук в
settings.json. Переключение мок → реальный сервер: endpoint+token в конфиге,
`rm state.json` → полная переотправка истории.

Проверка «каждые сутки» сидит в SessionStart (заход в Ленин), не на каждом
запросе: сессии длинные, launchd и так будит раз в день — чаще только шум.

## 8. Fragility — что может сломать и как контракт это держит

- **Ретрай после недоставленного ответа** (сервер принял, клиент не узнал) →
  дубль-чанк, случай `offset <` в §4: идемпотентно, ничего не задваивается.
- **Файл пересоздан/усечён** (size < offset у клиента) → клиент сбрасывает
  свой offset в 0 и шлёт заново; сервер ответит текущим размером (дубль) либо
  примет как новый.
- **Активная сессия дописывается во время чтения** → чанк режется по
  последнему `\n`, недописанная строка уедет следующим прогоном.
- **Двойной запуск** (launchd + хук одновременно) → flock на клиенте, второй
  процесс выходит.
- **Утечка токена** → отзыв (§6), токен скоупится одним owner/core, чужое не
  прочитать (ручка write-only, GET нет вовсе).
- **Часы клиента врут** → неважно: протокол не зависит от времени клиента,
  только от байтовых offset'ов.

## 9. Версионирование

`X-Uplink-Proto: 1` + `"proto"` в теле. Ломающие изменения → `/v2/...`,
клиент v1 должен продолжать работать до раздачи следующей обновы. Расширения
без слома (новые поля тела/ответа) — внутри v1, неизвестные поля игнорируются.
